# Environment diversity — brainstorm & evaluation

> **Date:** 2026-07-07 · **Status:** brainstorm / analysis (not a committed plan) · **Scope:** the game *environment* (maze structure, walls/floors, regions, atmosphere, wayfinding, obstacles) of `mazzzze`, and the `maze-gen` work it implies.
>
> **Language note:** this is a HOW/technical artifact → English (like `TECH_SPEC.md`). The ideas themselves are parked as terse Russian sparks under [`ideas/`](../../../ideas/README.md) as `IDEA-00NN`; this doc indexes and evaluates them.

This document is the evaluated companion to 30 new environment ideas (`IDEA-0027`…`IDEA-0056`) plus 4 pre-existing ones (`IDEA-0014/0021/0022/0023`). For each: **how to implement**, **where to get assets**, **story-point effort**, and **which side of the `mazzzze` / `maze-gen` boundary** the work falls on.

## 1. Why the current architecture makes this cheap

The environment render path today collapses everything to a two-value branch:

```
maze-gen RegionView.CellAt().IsPassable
   → MazeData.IsFloor / GetChunkData()   // int[,]  0 = floor, 1 = wall
      → Chunk.Setup() switch             // 0 → MeshLibrary item 0, 1 → item 1
         → MazeTiles.tres                // exactly 2 items (Floor, Wall)
```

Only `IsPassable` reaches the renderer; POIs and room membership are generated but discarded. That yields **four clean insertion seams**, and most ideas below hook one of them without touching maze generation:

| # | Seam | File | What it enables |
|---|------|------|-----------------|
| S1 | Richer cell query | `MazeData.GetChunkData` / new API | expose room/cave/POI/biome id instead of binary 0/1 |
| S2 | Tile selection | `Chunk.Setup` switch (`src/Chunk.cs:33-38`) | map a cell to one of *many* MeshLibrary items |
| S3 | Asset library | `MazeTiles.tres` | add floor/wall/prop variants beyond ids 0/1 |
| S4 | Per-chunk overrides | `ChunkManager.LoadChunk` | swap material / light / `WorldEnvironment` per zone |

**Two invariants any environment change must respect:**
1. Decorative scatter/props must be **non-blocking**, or update `IsFloor` in lockstep — the monster BFS (`Monster.FindPath`) and pickup line-of-sight pathfind over `MazeData.IsFloor`.
2. Structural changes must preserve **maze solvability** (one guaranteed entrance→exit path) and the deterministic-per-seed contract.

Almost every idea is verifiable via the existing headless-log / screenshot workflow (see `CLAUDE.md` → "How to verify changes").

## 2. maze-gen baseline (what already exists)

From `PlayersWorlds.Maps` (`../maze-gen`), consumed through the frozen `World` façade:

- **6 algorithms**: RecursiveBacktracker (long winding corridors), Aldous-Broder & Wilson's (uniform/unbiased), Hunt-and-Kill (winding, few long dead-ends), Sidewinder (horizontal bias), BinaryTree (open, diagonal bias). Pluggable via `RegionAlgorithm.Custom<T>()`.
- **4 recipe presets**: `Maze` (perfect), `Corridors` (Sidewinder), `Dungeon` (halls + corridors), `Caverns` (Aldous-Broder + caves).
- **Rooms/spaces**: `RoomKind.Hall` / `Cave` / `Blocked`, placed via `WithRooms(count, min, max, kind, tags…)`. **Already generated by the game today (2 Halls) but rendered as plain corridor.**
- **POIs**: `Entrance`, `Exit`, `DeadEnd` (longest-path ends + corridor tips). Adding a POI kind = a documented 3-step pattern (`PoiKind` enum → `RegionTags` const → compute in `World.Generate` + read in `RegionView.ComputePois`).
- **Fill factor** (0..1 → coarse enum), **cell/wall sizing** (corridor width / wall thickness).
- Low-layer cell tags exist but aren't surfaced as POIs: `MazeWallCorner`, `MazeVoid`. `RegionCell.Type` is currently uniform `Environment`.
- **Not present / stubbed**: braiding, weaving, elevation/3D (`NotImplementedException`), biome tagging (`AddEnvironmentAreas` is a no-op), seam-stitching between regions (Phase 2, reserved `Gates` only).
- **Known bugs to fix first**: longest-path start/end marker mis-tag; a `Serialize` trap. Both labeled trivial (S).

## 5. maze-gen roadmap delta (library-side work)

The library and game are separate concerns: where the game needs a world-building capability the library lacks, the **library and its integration façade are updated** rather than worked around. These items derive from the catalog and map onto maze-gen's existing phases (Phase 1 in progress → Phase 2 seams/store → Phase 3 richness). Complexity uses maze-gen's own S/M/L labels.

| Library work item | Serves | maze-gen phase | Size | Notes |
|-------------------|--------|----------------|------|-------|
| Fix longest-path mis-tag + `Serialize` trap | correctness for `0034` | Phase 1 (now) | **S** | prerequisite; already on the roadmap |
| Surface `RegionCell.Type` (Hall/Cave/Blocked) per cell | `0027`, `0028`, `0023` | Phase 3 (biome tagging) | **S–M** | data already exists internally; façade currently flattens to uniform `Environment` |
| New `PoiKind`s: LongCorridor / Junction / Plaza | `0035`, `0034`, `0040` | Phase 3 | **S** each | documented 3-step pattern; marks ride existing tag round-trip |
| Expose per-cell `DijkstraDistance` depth | `0034` | Phase 1/3 | **S** | computation already runs (`LongestTrailExtension`) |
| Growing Tree algorithm (mix-ratio param) | `0030`, `0029` | Phase 3 (recipe cookbook) | **S** | one `MazeGenerator` subclass via `Custom<T>` |
| Partial-braiding post-pass | `0031`, `0044` | Phase 3 | **S–M** | new `post_processing` extension; tunable % |
| Mask/outline shapes + sparse-fill voids surfaced | `0033` | Phase 3 | **M** | shape input + expose void cells as a distinct type |
| Environment/biome tagging (`AddEnvironmentAreas`) | `0014`, `0051` | Phase 3 | **M** | currently a no-op stub |
| Region-type dispatch (non-maze regions) | `0041` | Phase 3 / boundary | **M** | library defines the seam; forest/open generation is **game-owned** (maze-gen's stated boundary) |
| Cycle-injection support (optional) | `0044` | Phase 3 | **M** | game can also do this as a post-pass without library change |
| Seam-stitching between regions via `Gates` | `0042`, endless world | **Phase 2** | **L** | hardest; the load-bearing MMO capability |
| Elevation / true 3D geometry | `0043`, `0032` | Phase 3 | **L** | `NotImplementedException` today; primitives N-D-ready, concrete geometry hard-wired 2D |

**Reading:** the diversity payoff clusters in **Phase 3 (richness)**, and most of it is **S/M** — surfacing data that already exists (cell type, POI depth) and small additive passes (Growing Tree, braiding). The two **L** epics (seams, 3D) are genuinely hard and gate the "continue-the-world" and "verticality" ideas.

| Ambient / SFX | **Freesound** (CC0 filter), Kenney SFX | CC0 |

Legal rule of thumb: CC0 = zero obligation (default). CC-BY = keep `CREDITS.txt`. Avoid CC-BY-NC / CC-BY-SA / GPL art; marketplace "royalty-free" = bake into build, don't ship loose files.

## 7. Recommended build order (effort-ascending)

Grouped into sprints by ROI. Every item in Sprint 1–2 is Low complexity, additive, and headlessly verifiable — together they attack the three felt problems of a tall repeating maze at once: **perceived repetition**, **traversal frustration**, and **getting lost**.

**Sprint 1 — cheap atmosphere & variety (all G, ~13 SP):**
`0050` streaming fog · `0051` per-zone grading · `0052` dust particles · `0055` mesh/texture variants · `0037` breadcrumbs/leading-lines.

**Sprint 2 — traversal & identity (~17 SP):**
`0031` partial braiding (L) · `0034` dead-end loot + graded POIs (G+L) · `0014`+`0029` per-district identity + algorithm · `0036` skybox beacon/light shafts (G).

**Sprint 3 — content passes & obstacles (~23 SP):**
`0038` set-piece stamping · `0046` trap/hazard kit · `0027` rooms/caves as chambers (G+L) · `0053` reverb zones · `0035` landmark POIs (L).

**Sprint 4 — designed-dungeon feel (~27 SP):**
`0044` cyclic loops + lock/key · `0048` doors/gates · `0045` one-way shortcuts · `0040` vignettes · `0039` room prefabs.

**Milestones / defer (each its own project):**
`0032` weaving+bridges (13) · `0041` adjacent non-maze regions (13) · `0042` seams (21) · `0043` verticality (21) · `0049` moving platforms/water (8).

**Highest-ROI trio to start:** `0050`+`0051`+`0052` (atmosphere), `0031`+`0034` (maze feel), `0014`/`0029`+`0036` (wayfinding).

## 8. Scope / limits (границы)

- This is a **brainstorm/analysis**, not a committed plan. Nothing here is scheduled; SP are estimates for comparison, not commitments. Promoting any idea → a `requirements/REQ-NNNN/` with its own `design.md` (per `CLAUDE.md` rules), flipping the `IDEA` status to ✅ with a back-link.
- The **maze-gen / mazzzze boundary** is explicit: structure/topology (mazes, rooms, POIs, seams, 3D) is library work; look, atmosphere, obstacles, and **non-maze regions** (forests/open landmarks) are game work. §5 lists exactly what the library must gain.
- Ideas deliberately **excluded** as out-of-environment-scope: monster behaviour, item mechanics, HUD, camera, progression/meta — except where they touch the world (referenced, not designed here).
- Two hard constraints repeated for any implementer: keep decorative scatter **non-blocking** (monster BFS + pickup LoS read `IsFloor`), and preserve **solvability + per-seed determinism**.
