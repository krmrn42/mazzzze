## F-38: Использование системы переводов в коде

### Что локализуется, а что нет

| Тип строки | Локализуется? | Пример |
|------------|---------------|--------|
| Видимый пользователю текст UI | **Да** | «Рюкзак», «выбросить», имена предметов, подсказки |
| Строки с подстановкой значений | **Да** (локализуется шаблон, затем подставляются значения) | «Рюкзак  {N}/{M}» |
| Отладочные сообщения (`GD.Print`) | **Нет** | логи `[ChunkManager] LOAD …`, `[Player] Start cell=…` |
| Внутренние идентификаторы в данных | **Нет** | `ItemType = "vintage_camera"` (не показывается игроку) |

### Обращение к переводам

- Получение строки — стандартными средствами Godot: `Node.Tr("KEY")` (метод узла) либо `TranslationServer.Translate("KEY")` (статически, из любого места).
- Везде, где UI рисует/выводит текст, вызов идёт через перевод; «голых» строковых литералов в отображаемом тексте быть не должно.

### Строки с подстановкой (format strings)

- Локализуется **шаблон** с плейсхолдерами, затем значения подставляются форматированием.
- Шаблон в CSV: напр. ключ `INV_BACKPACK_COUNT` → `ru` «Рюкзак  {0}/{1}», `en` «Backpack  {0}/{1}».
- Порядок плейсхолдеров может различаться по языкам — поэтому используется **индексированная** подстановка (`{0}`, `{1}`, …), а не конкатенация.

### Соглашение об именовании ключей

`UPPER_SNAKE_CASE`, с префиксом домена/фичи:

| Префикс | Домен |
|---------|-------|
| `CORE_*` | ядро / общий UI |
| `MM_*` | мини-карта |
| `INV_*` | инвентарь |
| `ITEM_*` | предметы (имена, описания) |
| `CAM_*` | фотоаппарат |
| `PHOTO_*` | фотография |

Один ключ — одна смысловая строка.

### Миграция существующих строк

На момент постановки требования в коде есть видимые строки, требующие перевода:

| Где сейчас | Содержание | Предлагаемый ключ |
|------------|------------|-------------------|
| `src/InventoryHud.cs` (заголовок) | «Рюкзак  N/12» | `INV_BACKPACK_COUNT` |
| `src/InventoryHud.cs` (подсказка) | «Shift/G — выбросить» | `INV_DROP_HINT` |

При реализации эти литералы заменяются на вызовы перевода (шаблон + форматирование для счётчика), а значения заносятся в CSV.

### Будущие строки

Все новые UI-строки будущих фич должны заводиться через перевод с ключом в CSV сразу. В частности: имена предметов ([REQ-0013](../REQ-0013-vintage-camera/), [REQ-0017](../REQ-0017-photo/)), тексты состояния видоискателя и подсказка «входа» в фотографию.

> Конкретные приёмы (форматирование, bootstrap) — в [design.md](./design.md).
